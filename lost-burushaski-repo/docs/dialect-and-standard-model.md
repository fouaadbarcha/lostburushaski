# Dialect and Standard Model

## Major Dialects
The app should support three major dialects from day one:

- Hunza
- Nagar
- Yasin

These dialects should not be mixed together as if they are interchangeable.

## Key Rule
The same concept may exist across multiple dialects, but each dialect owns its own expression of that concept.

## Data Model Direction

### Concept Layer
A concept represents a shared meaning.

Example:
- anxiety
- restless feeling
- inner worry

### Dialect Entry Layer
Each dialect can have its own entry for the same concept.

Example:
- Hunza word
- Nagar word
- Yasin word

## User Experience Rules

### Default Dialect Preference
Users should choose or set a preferred dialect so the app can prioritize relevant entries.

### Cross-Dialect Visibility
The app should still allow users to see equivalents or related entries from other dialects when helpful.

### Ask Flow Must Be Dialect-Aware
Each question should specify:
- Hunza
- Nagar
- Yasin
- or not sure

## Standard Burushaski
Standard Burushaski is planned as a later curated learning layer.
It should not replace the dialects.

## Purpose of Standard Burushaski
- help non-native speakers learn
- provide a more consistent learning path
- support course building
- create a cleaner language onboarding experience

## Promotion Criteria for Standard Burushaski
A word can be promoted only if it is:
- widely understandable
- consistent
- teachable
- contextually accurate
- endorsed by trusted contributors and experts
- highly rated

## Important Constraint
Standard Burushaski should be curated, not automatically determined by popularity alone.
A high rating matters, but it should not override trust, accuracy, and teachability.

## Future Course Logic
The future course can be built from Standard Burushaski entries while still acknowledging dialect variants as legitimate forms used by native speakers.
